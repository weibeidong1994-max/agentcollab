# 多Agent协作管理系统 — 原始需求文档

## 1. 用户背景

本地电脑上安装了多个 AI Agent 工具，包括：

- **Claude Code**：通过 GLM Coding Plan Pro（¥149/月）接入，使用 GLM-5.1 / GLM-4.7 模型
- **KimiCode**：购买了 Moderato 套餐（¥99/月），使用 Kimi K2.5 模型
- **Hermes Agent**：NousResearch 出品的自进化 AI Agent，开源（MIT）
- **OpenClaw**：开源个人 AI 助理网关（MIT），支持 20+ 通信平台

开发量：轻量级开发，一周约 0.5-2 个项目。

## 2. 原始痛点

### 痛点 1：Agent 之间通信困难

当前各 Agent 独立运行，需要用户手动在多个窗口之间切换、反复转述上下文。Agent A 的输出需要用户复制粘贴给 Agent B，信息容易失真、遗漏约束。

### 痛点 2：部分 Agent 没有长期记忆

Claude Code 和 KimiCode 每次新会话从零开始，无法记住之前的项目决策、代码风格、架构约束。项目越长，上下文丢失越严重。

### 痛点 3：上下文难以查阅和维护

无法回头看"之前做到哪里了、为什么这样做、谁做的"。历史操作不可追踪，无法复盘和纠错。

### 痛点 4：换电脑后需要重新配置

项目上下文、Agent 配置、记忆数据无法方便地打包迁移到新电脑。

## 3. 原始期望

> 当我输入一个 Brief 的时候，这多个系统可以共同协作。

具体角色分工预期：

| Agent | 期望角色 |
|---|---|
| Claude Code | 写代码 |
| KimiCode | Review |
| Hermes | 后续的维护或者远程调度 |
| OpenClaw | 这个系统和我的对话 |

> 注：经沟通后调整，KimiCode 不适合做 Review（审查能力未被验证优于 Claude Code），改为辅助编码/额度分流；需求澄清由 AgentHub 工作台内置 LLM 完成，不依赖 OpenClaw；OpenClaw 仅用于产品上线后的应用层网关。

## 4. 核心需求

1. **每个 Agent 都有记忆，且共享 Agent 之间的记忆**，随时可以知道我做到哪里了。
2. **上下文和记忆可以查阅和维护**，支持检索、编辑、纠错、失效标记。
3. **可扩展**，后续有了新的 Agent，只要定义好角色，可以比较轻松地接入。
4. **可比较方便地打包和迁移**，项目可以整体导出并在新机器上恢复。

## 5. 套餐与额度约束

### 5.1 GLM Coding Plan Pro（¥149/月）

| 限制维度 | 额度 |
|---|---|
| 5 小时滚动窗口 | ~400 次 prompts |
| 每周 | ~2,000 次 prompts |
| GLM-5.1 消耗 | 高峰期（14:00-18:00）3 倍抵扣，非高峰期 2 倍抵扣 |
| GLM-4.7 消耗 | 1 倍（正常抵扣） |
| MCP 联网搜索+网页读取 | 1,000 次/月 |
| 使用约束 | 仅限交互式编程工具内使用，禁止自动化脚本/自定义后端/批量调用 |

### 5.2 KimiCode Moderato（¥99/月）

| 限制维度 | 额度 |
|---|---|
| 5 小时 token 配额 | 约 300-1,200 次 API 调用 |
| 每周 | 约 312 次 prompts（3 倍扩容后实测推算） |
| 最大并发 | 30 |
| 使用约束 | 同样禁止自动化脚本调用，仅限 Kimi CLI / Claude Code / Roo Code 内交互使用 |

### 5.3 额度消耗估算

一个轻量级项目约消耗 145-280 prompts，一周 2 个项目约 290-560 prompts/周。GLM Pro + Kimi Moderato 合计可以覆盖，但需精打细算，尤其是 GLM-5.1 高峰期 3 倍抵扣。

## 6. Agent 能力分析（搜索验证）

### 6.1 Claude Code

| 维度 | 详情 |
|---|---|
| 核心能力 | 多文件编辑、200K token 上下文、Git 深度集成、Subagent 并行执行、MCP 协议支持 |
| 优势 | 代码生成质量业界顶级；Subagent 可并行拆分子任务；MCP 生态扩展性好 |
| 关键短板 | **无跨会话持久记忆**——每次新会话从零开始；长会话内也存在上下文丢失 |
| 适合角色 | **主编码 + Review（通过 Subagent）** |

### 6.2 KimiCode

| 维度 | 详情 |
|---|---|
| 核心能力 | 256K 超长上下文（K2.5）、原生中文优化、文件读写与 Shell 执行、批量操作 |
| 优势 | 中文理解精准；256K 上下文对大项目友好；Agent Swarm 支持最多 100 个并行子 Agent |
| 关键短板 | 闭源且仅支持 Kimi 模型；无 LSP 支持；代码审查能力未被证明优于 Claude Code；执行速度较慢 |
| 适合角色 | **辅助编码 / 长上下文理解 / 额度分流**（不适合做 Review） |

### 6.3 Hermes Agent

| 维度 | 详情 |
|---|---|
| 核心能力 | 自学习闭环（Skill 自动生成与进化）、三层记忆、40+ 内置工具、6 种终端后端、Cron 定时任务、多平台通知、后台进程自动通知 |
| 优势 | 唯一具备原生持久记忆和自学习能力的 Agent；服务器部署不绑死笔记本；远程调度和自动化能力强 |
| 关键短板 | 自学习默认关闭；Skill 质量参差不齐需人工审查；无遗忘机制；**跨 CLI 编排能力尚未实现**（Issue #413 提案中） |
| 适合角色 | **远程调度 / 定时任务 / 后台维护 / 多平台通知**；未来可接管调度中心 |

### 6.4 OpenClaw

| 维度 | 详情 |
|---|---|
| 核心能力 | 多通道通信（20+ 平台）、三层记忆架构、700+ Skills 生态、Sub-agent 系统、定时调度 |
| 优势 | 记忆架构最完善；多通道通信覆盖最广；开源生态好 |
| 关键短板 | **不是编码工具**；Sub-agent 是隔离式设计，不支持协作；Skill 全量注入上下文浪费 Token |
| 适合角色 | **应用层网关**——产品上线后的常驻入口，而非开发时的调度器 |

## 7. 沟通中确定的关键决策

### 决策 1：委托模式

**选择**：不侵入任何 Agent 的内部运行方式，AgentHub 只做三件事——执行前注入上下文、执行后提取结果、Agent 之间做调度和传递。

**理由**：改造 Agent 内部意味着要跟四个不同厂商的更新节奏走，维护成本爆炸。

### 决策 2：KimiCode 角色调整

**选择**：从"Reviewer"调整为"辅助编码/长上下文理解/额度分流"。Review 由 Claude Code Subagent 承担。

**理由**：KimiCode 的代码审查能力未被验证优于 Claude Code，而 256K 上下文 + 原生中文优化更适合"理解大项目全貌"和"中文场景补充"。同时把简单编码任务分流到 Moderato 额度，让 GLM Pro 额度留给复杂任务。

### 决策 3：MVP 技术栈选择全 TypeScript

**选择**：MVP 阶段全栈 TypeScript（Node.js 后端 + Tauri 前端），不引入 Rust 和 Python。

**理由**：单人开发，一种语言比三种语言快得多；系统瓶颈在 Agent 的 API 响应时间（秒级），不在调度引擎的执行速度（毫秒级）；后续可按需局部引入 Rust。

### 决策 4：Hermes 做调度器，OpenClaw 做应用层网关

**选择**：Hermes 负责调度和维护，OpenClaw 负责产品上线后的常驻网关。

**理由**：Hermes 有原生记忆、Cron、多平台通知，天然适合调度；OpenClaw 的多通道通信和 Skill 生态天然适合做产品对外的入口。但 Hermes 当前跨 CLI 编排能力尚未实现，需分阶段推进。

### 决策 5：分阶段架构演进

| 阶段 | 调度方 | 说明 |
|---|---|---|
| 阶段 1（MVP） | AgentHub（TypeScript） | 轻量调度，通过 CLI 子进程调用各 Agent |
| 阶段 2 | Hermes Agent | 等 Issue #413 跨 CLI 编排实现后，Hermes 接管调度 |
| 阶段 3 | OpenClaw | 产品上线后，OpenClaw 作为应用层常驻网关 |

### 决策 6：额度感知调度

**选择**：AgentHub 的 Scheduler 必须具备额度感知能力，包括实时额度仪表盘、智能模型切换（高峰期切 GLM-4.7）、额度不足降级（切 KimiCode）、任务排队、成本预估。

### 决策 7：Coding Plan 合规调用方式

**选择**：AgentHub 通过启动 CLI 子进程（`claude -p` / `kimicode --prompt`）调用 Agent，不直接调 API。

**理由**：两个 Coding Plan 都禁止自动化脚本和自定义后端调用，但通过 CLI 子进程的交互式使用完全合规。

### 决策 8：原生记忆与共享记忆的关系

**选择**：AgentHub 共享记忆为权威源，Agent 原生记忆（Hermes MEMORY.md、OpenClaw MEMORY.md）为局部缓存，定期同步但以 AgentHub 为准。

### 决策 9：需求澄清由 AgentHub 自身完成

**选择**：需求澄清由 AgentHub 工作台内置 LLM 完成，不依赖 OpenClaw。

**理由**：AgentHub 本身就是用户和系统的对话界面，需求澄清是用户和系统之间的对话，不需要启动外部 Agent。在 AgentHub 内部完成澄清，信息天然进入共享记忆，无需额外提取和同步。OpenClaw 仅用于产品上线后的应用层网关。

## 8. 最终角色分工

| Agent | 角色 | 消耗来源 |
|---|---|---|
| AgentHub 自身 | 需求澄清 / 对话交互 | GLM Pro 额度（GLM-4.7，1 倍抵扣） |
| Claude Code | 主编码 + Review（Subagent） | GLM Pro 额度（非高峰 GLM-5.1 2 倍，高峰 GLM-4.7 1 倍） |
| KimiCode | 辅助编码 / 长上下文理解 / 额度分流 | Kimi Moderato 额度 |
| Hermes | 远程调度 / 定时任务 / 后台维护 / 多平台通知 | 自身服务（不消耗 Coding Plan） |
| OpenClaw | 应用层网关（仅产品上线后） | GLM Pro 额度（GLM-4.7，1 倍抵扣） |
