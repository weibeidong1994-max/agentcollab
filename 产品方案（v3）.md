# AgentHub — 产品方案（v3）

> 更新日期：2026-04-20
> 基于 v2 方案 + 第一天实际开发的经验总结和架构调整

---

## 1. 产品定位（修订）

**AgentHub** 是一个**本地优先的多 Agent 协作工作台**，核心定位从"任务编排器"调整为 **"上下文管家（Context Butler）"**。

一句话定义：

> 一个本地桌面应用，在嵌入式终端中管理多个 AI 编码 Agent 的交互会话，自动从对话中提取和沉淀上下文记忆，让 Agent 之间的知识可以流转。

### 1.1 与 v2 的核心差异

| 维度 | v2 设想 | v3 实际（经开发验证） |
|------|---------|---------------------|
| Agent 执行方式 | 后台 CLI 子进程 (`claude -p "prompt"`) | **嵌入式 PTY 终端**（用户可见、可交互） |
| 用户角色 | 输入 Brief → 审批计划 → 查看结果 | **直接在终端里和 Agent 对话**，系统自动记录 |
| 核心价值 | 任务编排和自动调度 | **上下文沉淀和跨 Agent 知识传递** |
| 记忆来源 | 用户手动输入 / LLM 提炼 | **自动从终端输出中提取** |
| 数据模型 | Brief → Task → Artifact → Memory | **Project → Conversation → Memory** |
| 技术栈 | 纯 TypeScript | **Tauri 2 (Rust) + React + TypeScript** |

### 1.2 为什么调整

v2 设想的"后台执行 + 自动编排"模式在实际开发中被用户否决。用户的真实需求是：

1. **我要看到 Agent 在做什么** — 不是黑盒执行，而是像自己打开终端一样
2. **我和 Agent 的对话就是上下文** — 不需要额外的 Brief 输入框
3. **系统自动帮我记东西** — 我只管对话，你负责提炼

## 2. 目标与边界

### 2.1 产品目标（修订后）

1. **统一入口**：在一个窗口内管理所有 Agent 的交互会话
2. **自动记忆蒸馏**：定时分析终端对话内容，提取用户需求、Agent 实现、文件操作、技术决策
3. **上下文文件化**：将提取的上下文写入项目目录的 `context.md` 文件
4. **一键注入**：让新启动的 Agent 通过读取 context.md 获得之前对话的上下文
5. **项目管理**：以项目为维度组织对话和记忆

### 2.2 非目标（明确不做）

1. **不替代 IDE**：不做成 VS Code 竞品，只是 Agent 交互的管理层
2. **不全自动调度**：不自动拆解任务、不自动分配 Agent、不自动审批
3. **不改 Agent 内部**：委托模式，只做外部包装
4. **不做云端同步**：纯本地运行
5. **不做多人协作**：单人工具

## 3. 设计原则（修订后）

1. **终端优先**：Agent 必须在嵌入式终端中运行，用户可以看到完整交互过程
2. **对话即上下文**：用户输入 + Agent 输出 = 原始上下文素材
3. **自动蒸馏**：系统定时从终端输出中语义提取关键信息，无需用户手动操作
4. **文件即记忆**：context.md 写入项目目录，Agent 可以直接读取
5. **委托模式**：不侵入 Agent 内部，只做外层包装
6. **最小侵入**：注入上下文时只发一行指令（`请阅读 context.md`），不粘贴长文本
7. **持久化**：客户端不关闭，对话就不结束；切换对话标签页时终端保持存活

## 4. 核心概念模型（修订后）

围绕三个核心对象：

```
Project（项目）
  ├── 一个文件夹 = 一个项目
  ├── 包含多个 Conversation
  └── 包含共享的 Memory

Conversation（对话）
  ├── 一次 Agent 交互会话
  ├── 绑定一个 Agent（Claude Code / KimiCode / ...）
  ├── 有独立的嵌入式终端
  └── 产生 Terminal Output → 自动蒸馏 → Memory

Memory（记忆）
  ├── 从终端输出中自动提取
  ├── 分类：decision / implementation / variable / feedback / general
  ├── 同时写入数据库 + context.md 文件
  └── 可人工删除/编辑
```

### 4.1 与 v2 概念模型的对比

| v2 概念 | v3 对应 | 变化原因 |
|---------|---------|---------|
| Brief | Conversation 的用户输入部分 | 不再单独收集 Brief，对话本身就是需求 |
| Task | 去掉 | 用户反馈不需要自动化任务拆解 |
| Artifact | 去掉 | Agent 的产物由 Agent 自己管理，不需要中间层 |
| Planner | 去掉 | 用户直接和 Agent 对话，不需要中间规划层 |
| Scheduler | 去掉 | 不做自动调度，用户手动选择 Agent |
| Context Builder | 简化为 context.md 文件生成 | 结构化 prompt 构建过度复杂 |
| Memory Service | 保留并简化 | 自动蒸馏代替手动/LLM 提取 |

## 5. 核心产品能力

### 5.1 项目管理

- 创建/切换项目，每个项目对应一个本地文件夹
- 项目下可以有多个对话（不同 Agent、不同主题）
- 左侧栏显示项目列表和对话列表

### 5.2 嵌入式终端（核心能力）

- 使用 xterm.js 渲染真实的 PTY 终端
- 支持 Claude Code / KimiCode / Hermes / OpenClaw 的完整 CLI 交互
- 终端内体验和原生 CLI 完全一致（颜色、光标、快捷键）
- 支持多对话并行（每个对话一个独立终端实例）
- 切换对话标签页时终端保持存活（不销毁重建）

### 5.3 自动蒸馏（核心能力）

- 每 10 秒扫描一次所有活跃终端的输出
- 从原始终端文本中模式匹配提取：
  - **用户需求**：匹配中文意图词（帮我/请/需要/想要/实现...）、Claude Code 的 `›` 提示符后的文字
  - **文件操作**：匹配 `Wrote/Created/Modified X lines to filename`
  - **Agent 执行内容**：匹配列表项（●/•/数字序号）
  - **技术决策**：匹配决策关键词（决定/选择/采用/确定...）
  - **涉及文件**：匹配文件扩展名路径
- 提取结果去重后写入 SQLite 数据库
- 同时生成/更新 `{project_path}/.agenthub/context.md`

### 5.4 记忆面板

- 中间栏显示当前项目的所有记忆
- 按分类着色：决策(黄)/实现(蓝)/变量(绿)/反馈(紫)/通用(灰)
- 支持单条删除
- 显示 context.md 文件路径
- 一键复制全部上下文到剪贴板

### 5.5 上下文注入

- 点击「注入上下文」按钮
- 向当前终端发送：`请阅读项目上下文文件: {path}`
- Agent 读取 context.md 文件获得之前的对话上下文
- **不是粘贴长文本**（避免 token 浪费和格式混乱）

### 5.6 Agent 管理

- 启动时自动检测已安装的 Agent
- Health Check 验证 Agent 是否可用
- 每个 Agent 有独立的配置（命令路径、参数、环境变量）

## 7. 关键用户流程

### 7.1 典型使用流程

```
1. 打开 AgentHub
2. 创建项目（或选择已有项目）
3. 创建对话 → 选择 Agent（如 Claude Code）
4. 在嵌入式终端中直接和 Agent 对话（就像用原生 CLI 一样）
5. 系统每 10 秒自动从终端输出提取上下文 → 写入数据库 + context.md
6. 记忆面板实时显示提取结果
7. 开启第二个对话 → 选择另一个 Agent（如 KimiCode）
8. 点击「注入上下文」→ 第二个 Agent 读取 context.md 获得第一个对话的上下文
9. 两个 Agent 基于共享上下文协作
```

### 7.2 上下文流转

```
用户和 Agent A 对话
  → 终端输出持续累积
  → 定时蒸馏引擎扫描输出
  → 模式匹配提取: 用户需求 / 文件操作 / 执行内容 / 技术决策
  → 写入 SQLite (memories 表)
  → 生成 context.md 到 {project}/.agenthub/context.md
  
用户启动 Agent B
  → 点击「注入上下文」
  → 发送: "请阅读项目上下文文件: {path}"
  → Agent B 读取 context.md
  → Agent B 获得之前对话的全部上下文
```

## 9. 已验证的决策点（来自今天开发实践）

### 9.1 架构相关

| # | 决策 | 结论 | 验证方式 |
|---|------|------|---------|
| D1 | 终端嵌入 vs 后台执行 | **必须嵌入式终端** | 用户明确要求看到 Agent 执行过程 |
| D2 | PTY 实现 | **Rust portable-pty** | Tauri Shell.spawn() 不提供 PTY |
| D3 | 终端渲染 | **xterm.js** | 业界标准，支持 ANSI/颜色/光标 |
| D4 | 数据库 | **tauri-plugin-sql (SQLite)** | WebView 不能用 Node.js 模块 |
| D5 | 文件读写 | **Rust 自定义命令** | write_file/read_file/mkdirp |
| D6 | 会话持久化 | **客户端不关闭就不销毁** | key={} 强制 remount 导致丢失 |

### 9.2 蒸馏相关

| # | 决策 | 结论 | 验证方式 |
|---|------|------|---------|
| D7 | 轮次追踪 vs 原始输出 | **原始输出模式匹配** | 轮次追踪对 TUI Agent 完全不可靠 |
| D8 | 正则提取 vs 行过滤 | **多维度模式匹配** | 单一正则无法覆盖所有场景 |
| D9 | 注入方式 | **文件引用而非长文本** | 避免 token 浪费和格式混乱 |
| D10 | 蒸馏频率 | **10 秒** | 平衡实时性和性能 |
| D11 | 记忆存储 | **数据库 + 文件双写** | 数据库用于 UI 展示，文件用于 Agent 读取 |

### 9.3 UX 相关

| # | 决策 | 结论 | 来源 |
|---|------|------|------|
| D12 | 不需要手动完成按钮 | **自动持续蒸馏** | 用户不想手动触发 |
| D13 | 不需要手动摘要输入 | **自动提取** | 用户只想对话，不想填表 |
| D14 | 记忆要能看到存在哪 | **显示文件路径** | 用户反馈"不知道保存在哪" |
| D15 | 切换对话不丢失终端 | **隐藏而非销毁** | 用户反馈"切出去再切回来记录消失" |

## 10. 当前限制和已知问题

### 10.1 蒸馏质量

当前使用**基于规则的模式匹配**提取上下文，不是真正的语义理解：

- 能正确识别：`Wrote X lines to file.html`、`› 帮我写xxx`、`● 完成了xxx`
- 可能遗漏：Agent 用非标准格式输出的信息
- 可能误判：代码行被当作"实现内容"
- **未来改进方向**：调用 LLM API 做真正的语义总结（可用 GLM API）

### 10.2 终端持久化

当前使用 `key={convId}` 强制 TerminalPanel 重新挂载，切换对话会丢失终端状态。

- **待实现**：渲染所有 TerminalPanel 但 CSS 隐藏非活跃的
- 这需要在 App.tsx 中改变渲染策略

### 10.3 多对话并行

当前蒸馏逻辑遍历所有 terminalRefs，支持多对话同时蒸馏。
但 UI 上同一时间只显示一个终端，其他终端虽然存活但不可见。

## 12. 成功指标（修订后）

### 北极星指标

**用户在一次完整的"对话-蒸馏-注入-再对话"循环中的满意度**

### 关键衡量指标

1. 蒸馏准确率：提取的记忆中有多少是真正有价值的
2. context.md 可读性：Agent 读取 context.md 后能否理解之前做了什么
3. 终端体验：嵌入式终端是否和原生 CLI 一致（延迟、渲染、快捷键）
4. 记忆覆盖率：一次对话结束后，关键信息是否有遗漏
