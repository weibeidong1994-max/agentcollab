# 多Agent协作管理系统 — 产品方案（v2）

## 1. 产品定位

产品暂定名：`AgentHub`

`AgentHub` 是一个本地优先的多 Agent 协作工作台，作为统一入口、统一记忆层、统一编排层和统一审计层，把 `OpenClaw`、`Hermes Agent`、`KimiCode`、`Claude Code` 等能力组织起来，让它们围绕同一个 Brief 持续协作。

一句话定义：

> 一个本地优先、可扩展、可迁移、可审计、额度感知的多 Agent 协作 IDE。

## 2. 目标与边界

### 2.1 产品目标

1. 降低多 Agent 协作成本：用户不再手动来回转述上下文。
2. 为弱记忆 Agent 补足长期记忆：让 `Claude Code`、`KimiCode` 这类原生无长期记忆的 Agent 具备"看起来有记忆"的能力。
3. 建立可维护的上下文体系：任务、决策、约束、偏好、产物都能被查阅、纠错、回滚。
4. 建立可持续扩展的接入机制：新增 Agent 时尽量只配置角色与适配器，不改动主系统。
5. 额度感知与成本控制：在有限的 Coding Plan 额度内智能调度，避免额度浪费。

### 2.2 非目标

1. 训练自己的基础模型。
2. 完全替代原生 IDE。
3. 一开始就支持多人实时协作。
4. 一开始就做云端强依赖架构。
5. 侵入式改造任何 Agent 的内部运行方式。

## 3. 设计原则

1. **本地优先**：核心数据、任务编排、记忆存储、日志审计默认都可离线运行。
2. **人在回路**：涉及写文件、执行命令、部署上线、覆盖记忆等动作时必须可审批。
3. **委托模式**：不侵入 Agent 内部，只做执行前注入上下文、执行后提取结果、Agent 之间做调度和传递。
4. **薄编排、厚上下文**：编排层负责组织，真正提升协作效果的是可维护的共享上下文。
5. **结构化优先**：Agent 之间传递的不只是自然语言，还包括任务卡、产物、状态、证据。
6. **可替换**：任意单个 Agent 都可以新增、替换、下线，不绑死某个供应商。
7. **可追溯**：任何结果都能追溯"是谁在什么上下文下做出的"。
8. **额度感知**：调度决策必须考虑 Coding Plan 额度消耗，智能选择模型和 Agent。

## 4. 核心用户与典型场景

### 4.1 目标用户

1. 单人开发者：本地装有多个 AI 编码工具，希望它们分工协作。
2. 技术负责人：希望把设计、编码、Review、部署、维护分给不同 Agent。
3. AI 重度用户：对上下文复用、长期记忆、任务历史和迁移有强需求。

### 4.2 典型场景

1. **新功能开发**：AgentHub 工作台澄清需求，`Claude Code` 编码，`Claude Code Subagent` Review，`Hermes` 部署与通知。
2. **缺陷修复**：一个 Agent 复现问题，一个 Agent 读日志定位，一个 Agent 生成修复，一个 Agent 回归检查。
3. **长周期项目维护**：系统记住架构决策、代码风格、部署环境和历史坑点。
4. **跨机器迁移**：将项目上下文、任务历史、适配器配置迁移到新电脑继续工作。
5. **产品上线后常驻服务**：`OpenClaw` 作为产品对外的多通道网关，用户通过 Telegram/微信等与产品交互。

## 5. 角色分工与调度策略

### 5.1 角色定义

| Agent | 角色 | 消耗来源 | 说明 |
|---|---|---|---|
| AgentHub 自身 | 需求澄清 / 对话交互 | GLM Pro（GLM-4.7，1 倍抵扣） | 工作台内置 LLM 对话，无需启动外部 Agent |
| Claude Code | 主编码 + Review（Subagent） | GLM Pro（非高峰 GLM-5.1 2 倍，高峰 GLM-4.7 1 倍） | 代码生成质量最强；Subagent 可做 Review |
| KimiCode | 辅助编码 / 长上下文理解 / 额度分流 | Kimi Moderato | 256K 上下文 + 中文优化；分流简单编码任务节省 GLM 额度 |
| Hermes | 远程调度 / 定时任务 / 后台维护 / 多平台通知 | 自身服务 | 原生记忆 + Cron + 多平台通知；未来可接管调度中心 |
| OpenClaw | 应用层网关（仅产品上线后） | GLM Pro（GLM-4.7，1 倍抵扣） | 多通道通信和 Skill 生态适合做产品对外的入口 |

### 5.2 额度感知调度策略

| 优先级 | 场景 | 用哪个 Agent + 模型 | 消耗来源 |
|---|---|---|---|
| 1 | 简单编码/批量修改 | KimiCode（K2.5） | Moderato 额度 |
| 2 | 复杂编码（非高峰期） | Claude Code + GLM-5.1 | Pro 额度（2 倍） |
| 3 | 复杂编码（高峰期 14:00-18:00） | Claude Code + GLM-4.7 | Pro 额度（1 倍） |
| 4 | Review | Claude Code + GLM-4.7 | Pro 额度（1 倍） |
| 5 | 需求澄清 | AgentHub 内置 + GLM-4.7 | Pro 额度（1 倍） |
| 6 | 部署/通知 | Hermes | 免费 |

### 5.3 能力重叠决策规则

当多个 Agent 能做同一件事时，按以下顺序决策：

1. 额度优先：优先使用额度充裕的套餐。
2. 能力匹配：选择在该任务类型上表现最好的 Agent。
3. 可用性检查：确认 Agent 已安装且健康。
4. 权限匹配：确认 Agent 有执行该任务的权限。
5. 成本考量：同等条件下选择消耗更低的方案。

### 5.4 Agent 不可用时的降级策略

1. 同角色备选 Agent 接管。
2. 人工接管。
3. 任务挂起等待 Agent 恢复。

## 6. 核心概念模型

系统围绕六个核心对象运转：

1. `Project`：一个项目工作空间，承载代码、记忆、任务、配置。
2. `Brief`：用户输入的原始意图。
3. `Task`：由 Brief 拆解出的执行单元，包含目标、输入、依赖、验收标准。
4. `Artifact`：任务产生的结构化产物，如代码变更、Review 报告、设计文档、部署结果。
5. `Memory`：沉淀下来的长期上下文，包括事实、偏好、约束、决策、经验。
6. `Agent Profile`：一个 Agent 的角色、能力、限制、调用方式、权限边界、额度来源。

关系链：`Brief -> Task -> Artifact -> Memory -> 下一轮 Task`

## 7. 核心产品能力

### 7.1 统一入口

用户只和一个入口交互：

1. 输入 Brief。
2. 系统在工作台内通过 LLM 对话澄清需求（信息不足时追问）。
3. 查看系统拆解出的任务计划（含预估额度消耗）。
4. 决定是否批准执行。
5. 跟踪各 Agent 进度与结果。
6. 对记忆和上下文进行修正。

### 7.2 共享记忆系统

分层管理：

1. 用户级记忆：长期偏好，如语言、代码风格、沟通方式。
2. 项目级记忆：技术栈、架构约束、环境信息。
3. 任务级记忆：当前阶段目标、依赖、风险、阻塞点。
4. Agent 级记忆：某个 Agent 的运行习惯、特定提示模板、失败恢复信息。

记忆必须支持：自动提取、人工确认、可检索、可编辑、可失效、可追溯来源。

**原生记忆协调策略**：AgentHub 共享记忆为权威源，Agent 原生记忆（Hermes MEMORY.md、OpenClaw MEMORY.md）为局部缓存，定期同步但以 AgentHub 为准。

### 7.3 任务编排与协作

支持三种协作方式：

1. 串行：设计 -> 编码 -> Review -> 部署。
2. 并行：测试、文档、审计并发执行。
3. 回环：Review 驳回后自动回到编码阶段，直到满足验收标准。

编排基于"角色能力 + 当前可用性 + 权限范围 + 额度状态"做分配，不写死 Agent 名字。

### 7.4 上下文管理台

1. 查看每次任务注入给 Agent 的上下文。
2. 标记哪些记忆是可信的、过期的、待确认的。
3. 手动修正错误记忆，避免错误长期污染。
4. 对比不同任务阶段的上下文快照。

### 7.5 Agent 注册与扩展

新增 Agent 时至少要定义：角色、能力标签、调用方式、输入输出协议、权限范围、故障处理策略、额度来源。

如果新 Agent 满足标准 CLI 协议，则只需配置即可接入；如果不满足，则补一个适配器。

### 7.6 审批与安全控制

以下动作默认进入审批队列：

1. 批量改文件。
2. 执行 Shell 命令。
3. 安装依赖。
4. 提交代码。
5. 推送远端。
6. 部署和访问敏感环境。
7. 覆盖关键记忆。

**工作区锁定机制**：同一文件同一时刻只允许一个 Agent 写入，避免并发冲突。

### 7.7 额度仪表盘与成本管理

1. 实时显示 GLM Pro 和 Kimi Moderato 的剩余额度（5h 窗口 + 周额度）。
2. 智能模型切换：根据当前时段自动选择 GLM-5.1（非高峰）或 GLM-4.7（高峰）。
3. 额度不足降级：当 GLM Pro 额度紧张时，自动把编码任务切到 KimiCode。
4. 任务排队：5h 额度耗尽时，非紧急任务自动排队等待额度恢复。
5. 成本预估：在用户审批执行计划时，展示预估的 prompt 消耗。

### 7.8 打包与迁移

迁移包至少应包含：

1. 项目代码和快照。
2. 项目级记忆。
3. 任务历史与产物。
4. Agent 配置。
5. 版本清单与依赖指纹。
6. 恢复向导所需的兼容性信息。

密钥和本地敏感凭据不直接打包，而是通过恢复向导重新绑定。

## 8. 关键用户流程

### 8.1 从 Brief 到结果

1. 用户输入 Brief。
2. 系统在工作台内通过 LLM 对话澄清需求（信息不足时追问，澄清过程天然进入共享记忆）。
3. 系统补全相关上下文并生成计划草案（含额度预估）。
4. 用户确认执行范围和风险动作。
5. 系统按 DAG 分发任务给不同 Agent。
6. 各 Agent 输出结构化产物。
7. 系统抽取记忆、更新状态、生成下一步上下文。
8. 用户在统一界面查看结果、追问、回退或继续推进。

### 8.2 Review 驳回回环

1. Claude Code 编码 Agent 生成代码产物。
2. Claude Code Subagent 输出问题清单和修改建议。
3. 调度器把问题转成新的返工任务。
4. 原编码 Agent 或 KimiCode 接手修改。
5. 再次进入 Review，直到通过。

### 8.3 记忆纠错流程

1. 用户发现错误记忆。
2. 在上下文管理台中编辑或标记失效。
3. 系统保留版本记录。
4. 后续任务不再注入旧记忆。
5. 同步更新到相关 Agent 的原生记忆文件。

## 9. 信息架构

主界面由六个主导航组成：

1. `工作台`：输入 Brief、LLM 对话澄清需求、查看计划、审批执行。
2. `任务`：看 DAG、依赖、状态、失败原因、重试。
3. `记忆`：查阅和维护共享记忆。
4. `产物`：查看代码、报告、文档、部署结果。
5. `Agent`：管理接入的 Agent、权限、健康状态和角色配置。
6. `额度`：查看各套餐剩余额度、消耗历史、成本预估。

## 10. 分阶段架构演进

### 阶段 1（MVP）：AgentHub 轻量编排 + Hermes 记忆补充

AgentHub（TypeScript）做调度，通过 CLI 子进程调用各 Agent。Hermes 作为"执行 Agent + Cron + 通知"，其记忆系统作为 AgentHub 共享记忆的补充数据源。

### 阶段 2：Hermes 接管调度

等 Hermes Issue #413（跨 CLI 编排）实现后，AgentHub 的调度逻辑迁移到 Hermes Skill。AgentHub 退化为"上下文管理台 + 记忆治理 UI"。Hermes 成为事实上的调度中心。

### 阶段 3：OpenClaw 接入应用层

产品上线后，OpenClaw 作为产品的对外多通道网关。用户通过 Telegram/微信/Slack 和产品交互。OpenClaw 的 Skill 作为产品的能力扩展。

## 11. MVP 范围

### 11.1 首版必须做

1. 单机本地运行。
2. 统一 Brief 输入 + 工作台内 LLM 需求澄清。
3. 至少接入 `Claude Code`、`KimiCode`、`Hermes` 三类角色（OpenClaw 留到产品上线后）。
4. 支持串行工作流：需求澄清 -> 编码 -> Review。
5. 支持项目级共享记忆。
6. 支持任务历史、日志、产物留存。
7. 支持高风险动作审批。
8. 支持额度仪表盘（至少显示 GLM Pro 和 Kimi Moderato 剩余额度）。
9. 支持智能模型切换（高峰期自动切 GLM-4.7）。

### 11.2 首版可以不做

1. 云端同步。
2. Agent 市场。
3. 多人协作。
4. 复杂插件生态。
5. 完整部署自动化。
6. Hermes 跨 CLI 编排（等 Issue #413）。
7. OpenClaw 应用层网关（等阶段 3）。

## 12. 成功指标

### 12.1 北极星指标

在一个真实开发任务中，用户完成"输入 Brief 到交付结果"的平均人工介入次数下降。

### 12.2 关键衡量指标

1. 单任务重复补充上下文次数。
2. Agent 任务交接成功率。
3. Review 驳回后的自动返工成功率。
4. 共享记忆命中率。
5. 用户手动纠正记忆的频次。
6. 新 Agent 平均接入时间。
7. 项目迁移后恢复成功率。
8. 额度利用率（实际消耗 / 可用额度）。
9. 高峰期额度耗尽次数。

## 13. 版本路线图

### P0：跑通闭环

目标：先证明"共享记忆 + 串行协作 + 额度感知"成立。

范围：CLI 或极简桌面界面、统一任务计划、四类角色接入、项目级记忆检索与注入、Review 返工回环、额度仪表盘。

### P1：可日常使用

目标：让系统真正替代手工切换流程。

范围：完整桌面 GUI、上下文浏览与记忆维护、审批中心、失败重试和任务恢复、基础导入导出、智能模型切换。

### P2：可扩展可迁移

目标：降低接入和迁移成本。

范围：标准化 Agent Profile、通用 CLI 适配器、迁移恢复向导、模板化工作流、更细粒度权限控制。

### P3：架构演进

目标：Hermes 接管调度，OpenClaw 接入应用层。

范围：Hermes 跨 CLI 编排集成、调度逻辑迁移到 Hermes Skill、OpenClaw 应用层网关配置、AgentHub 退化为 UI 层。

## 14. 产品决策摘要

1. 优先做本地单机，而不是一开始做云协作。
2. 优先解决"上下文"和"协作交接"，而不只是多开几个 Agent。
3. 优先支持结构化任务卡和产物，而不是纯聊天中转。
4. 优先做记忆治理能力，而不是把所有历史都当作记忆。
5. 优先做人可控的执行系统，而不是完全自动驾驶。
6. 委托模式——不侵入 Agent 内部，只做注入、提取、调度。
7. KimiCode 做辅助编码和额度分流，不做 Review。
8. MVP 全 TypeScript，不引入 Rust 和 Python。
9. Hermes 做调度和维护，OpenClaw 仅做应用层网关（产品上线后），分阶段推进。
10. 额度感知是核心能力，不是锦上添花。
11. 需求澄清由 AgentHub 工作台内置 LLM 完成，不依赖外部 Agent。
